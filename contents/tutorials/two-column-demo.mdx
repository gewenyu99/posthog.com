# JavaScript Module Patterns

Let's explore different ways to export and import JavaScript modules using practical examples.

## Default Exports

The most common pattern is using default exports for your main module functionality. The `PostHogClient` class is exported as the default export, making it the primary export of the module. This is commonly used for the main functionality of a module.
</DemoReference>

## Importing Default Exports

<DemoReference
  file="code-examples/two-column-demo/b.js"

To use a default export, we use a simple import statement without curly braces. The name can be anything we choose, though it's common to use the same name as the export.
</DemoReference>

## Named Exports

Named exports allow us to export multiple values from a module. Here we export constants and utility functions that can be imported individually. This is useful for exporting multiple related pieces of functionality.
</DemoReference>

<DemoReference
  file="code-examples/two-column-demo/b.js"

## Importing Named Exports

Named exports are imported using curly braces, allowing us to select exactly which exports we need. We can also rename them during import using the `as` keyword.
</DemoReference>

## Advanced Export Patterns

JavaScript modules support more advanced patterns like renaming exports and using computed properties. These patterns provide flexibility in how we structure and expose our module's functionality.
</DemoReference>

## Advanced Import Patterns

We can import everything as a namespace using `* as`, or use dynamic imports for code splitting. Dynamic imports return a promise, allowing us to load modules on demand.
